# 2026-06-16 F1-M3 DMA — DmaPool（批2）

## 背景

批1 落地 DmaBuffer（move-only 句柄）。批2 提供 DmaPool：把 PMM（物理页分配）+ VMM（映射）封装成 `alloc(size) → ErrorOr<DmaBuffer>`，注入 release 回调实现 RAII 自动归还。下游（F5-M1 AHCI）契约：`g_dma_pool.alloc()` 拿到带 phys/virt 的 buffer。

## 目标

DmaPool.alloc 端到端可用：分配物理页、建立 CPU 可访问映射、返回自释放 DmaBuffer；显式 free + RAII 均归还页；OOM / size==0 报 Error。

## 设计

### phys→virt：复用 direct-map（virt = phys + KERNEL_VMA）

kernel 有 higher-half direct map（`KERNEL_VMA = 0xFFFFFFFF80000000`，phys + KERNEL_VMA = virt，物理内存的固定窗口）。DmaPool 复用它：alloc 拿 phys，virt = phys + KERNEL_VMA，`vmm.map` 确保 PTE 存在。**virt 由 phys 唯一决定**，无需 virt 地址分配器、不会泄漏 virt 空间。

### 对齐

PMM 按页（4 KiB）分配，页对齐满足所有常见设备对齐（AHCI cmd table 128 B、FIS 256 B、PRDT 4 KiB）。请求向上取整到整页（256 B 请求消耗整页）。

### release 回调注入

alloc 返回的 DmaBuffer 带 `&DmaPool::release_callback`（批1 的 `DmaReleaseFn` 函数指针）。析构/重赋值自动 `free_pages(phys)`。`free()` 显式早释放。

## 关键陷阱（致命，值得长记）

### free 绝不能 unmap direct-map 窗口

首版 `return_pages` 做 `vmm.unmap(virt)`（virt = phys + KERNEL_VMA）——这**拆掉了 direct-map 的一个槽**。direct-map 是 phys↔virt 的**永久固定映射**、整个 kernel 共用（访问物理内存、遍历页表都靠 `phys_to_virt = phys + KERNEL_VMA`）。更糟：**direct-map 是 demand-paged**（boot 不全映射，首次访问才建 PTE）。

后果链：
1. unmap 后该 phys 的 direct-map 槽空。
2. 后续任何代码（fork/usermode 页错误处理、数据访问）访问该物理地址 → page fault。
3. demand-page handler 把该 virt 映射到**随机新 phys**（不是原 phys）→ 数据不一致 + 该新 phys 又被间接访问 → 再次 fault。
4. 同一 virt（如 `0xFFFFFFFF82102000`）反复 demand-page，每次新 phys 递增，**永不收敛 → 死循环**，QEMU 卡死。

run-kernel-test 表现：DmaPool 测试之后进入 usermode 时无尽 `[VMM] Demand-paged 0xFFFFFFFF82102000 -> phys 0x2103000/2105000/...`，QEMU 挂起。另有 `test_free_count` FAIL（alloc 的 map walk 触发 demand-page 消耗页表页，`free_page_count != before-2`）。

**修复**：`return_pages` 只 `free_pages(phys)`，**绝不 unmap**。direct-map PTE 永久（phys↔virt 固定语义）；phys 回收后可重分，若再作 DMA buffer，alloc 的 map 幂等覆盖同 PTE。这正是 AHCI 驱动的安全模式（map cmd list/FIS 但从不 unmap）。

### direct-map demand-paged → free_count 不对称

direct-map demand-paged 意味着 alloc 的 `map` walk（walk_level 经 `phys_to_virt` 访问中间页表）可能触发 demand-page、分配**共享页表页**。这些页表页 free 时不回收。所以 `free_page_count` 不会精确回到 alloc 前。

测试调整：`test_free_count` 不再断言 `== before`，改断言方向（alloc 消耗、free 回收数据页）。`test_raii` 不测 `translate==0`（按设计不 unmap），改测 free_count 回收。

## 验证

run-kernel-test **681 → 687**（+6）：alloc 映射 + translate 一致、CPU 读写 round-trip、显式 free 回收页、RAII 析构回收页、OOM→OutOfMemory、size==0→InvalidArgument。死循环消失。
