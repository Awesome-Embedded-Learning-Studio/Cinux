# 2026-07-05 PIE 批2 — toolchain 去 -fno-pie，gcc/g++ 默认 PIE

## 背景

批1([note](2026-07-05-pie-b1.md))让内核支持 PIE main(ET_DYN + ASLR base)。批2 把 toolchain 的 `-fno-pie -no-pie` 拿掉,放 gcc/g++ 默认产 PIE(现代 distro 的 `--enable-default-pie`),端到端跑通「CinuxOS 上 `gcc /hello.c` → PIE a.out → 跑起来」。

## 改动(B2,3 文件)

- [extract.sh](../tools/gcc-toolchain/extract.sh):去掉 `gcc -S`、`gcc -H`、`g++ -H` 三处的 `-fno-pie`(line 173/183/278)+ 注释更新。gcc 默认 PIE → hello.s 是 RIP-relative 汇编,头闭包不变(PIE/ET_EXEC 头路径一样)。
- [cinux-usability-test.sh](../rootfs/overlay/etc/cinux-usability-test.sh):`gcc -fno-pie -no-pie /hello.c` → `gcc /hello.c`;`g++ -fno-pie -no-pie -fno-use-linker-plugin` → `g++ -fno-use-linker-plugin`(保留 `-fno-use-linker-plugin`,那是 lto_plugin `invalid ELF` 的 work-around,与 PIE 无关)。
- [assemble_gcc_rootfs.sh](../scripts/assemble_gcc_rootfs.sh):头部注释更新。

**保留不动**:`tools/musl/build-hello.sh` / `build-cinux-exit.sh` / `build-forktest.sh` 的 `-static -no-pie`。这些是**静态 ET_EXEC** 测试二进制(有意,验证 kernel 静态加载路径),不是 PIE 目标。静态 PIE(`-static-pie`)是另一回事,留 follow-up。

## ⭐陷阱:GUI=ON 让 usability gate 卡死

`run-buildroot-usability`(isa-debug-exit gate)在 `CINUX_GUI=ON` 下**卡到 timeout**:
```
[INIT] GUI worker thread launched
[SCHED] Task tid=3 'kernel_init' exited
[GUI] Worker thread started
...(timeout 杀,没跑 busybox init / cinux-usability-test.sh)
```

根因:`CINUX_GUI=ON` 让 `kernel_init` 走 **GUI desktop** 初始化路径(启动 gui_worker 后 exit),**不 exec busybox init PID1**,所以 `inittab::once` 的 `cinux-usability-test.sh` 永远不跑,isa-debug-exit 不触发,gate 死等。

`CINUX_GUI=ON` 适合 `cmake --target run`(GUI 预览,用户点 Shell 手动操作);**usability gate 必须 `CINUX_GUI=OFF`**(console 走 busybox init → test script → cinux-exit)。两个模式用途不同,不能混。

(日志里的 `#BP (vector 3)` 是 kernel 自测的 int3 异常处理测试 `[TEST] Triggering breakpoint exception`,**不是崩溃**,别误判。)

## 验证

`CINUX_GUI=OFF` + 重新 assemble gcc rootfs(extract.sh 默认 PIE)后,`run-buildroot-usability` **PASS**:
```
[EXECVE] loaded /tmp/a.out  base=0x000000002024D000 (dynamic)  ← gcc PIE main
Hello from GCC!
[usability] PASS gcc-compile-run

[EXECVE] loaded /tmp/cpp.out base=0x00000000209E4000 (dynamic)  ← g++ PIE main
Hello from G++!
exception: caught
[usability] PASS gpp-compile-run

[usability] result: PASS
```

关键点:
- a.out / cpp.out 都是 **ET_DYN PIE**,base 落在 `[0x20000000, 0x21000000)`(USER_EXEC_BASE + ASLR);**两次 base 不同**(0x2024D000 / 0x209E4000)→ main-base ASLR 生效。
- a.out 用 **glibc ldso**(`/lib64/ld-linux-x86-64.so.2 base=0x10000000`),走批1 的 PIE 路径(与 musl ldso 的 hello-dyn 同机制,不同 ldso)。ldso 重定位 main 后跳 `AT_ENTRY = main_base + e_entry`。
- `[EXECVE] loaded /sbin/cinux-exit base=0x0` —— cinux-exit 是静态 ET_EXEC,走批1 保留的 ET_EXEC 兼容路径(main_base=0),不受影响。
- 残留 `unhandled syscall 273/334/302/318`(rseq/getcpu 等 glibc probe)走 ENOSYS fallback,非致命(g++ 仍 PASS)—— 见 [[gcc-missing-syscalls]]。

## 结论

PIE 弧(B1+B2)收官:CinuxOS 现在完整支持 PIE main(ET_DYN + ELF-base ASLR),gcc/g++ 默认产 PIE 跑通,ET_EXEC 兼容路径不破坏。对齐 Linux 完整 ASLR(stack/mmap/brk/main-base 四处)。

## Follow-up

- **interpreter ASLR**:ldso 仍固定 `USER_INTERP_BASE`(256 MB),未随机化(与 main base 配对的 follow-up)。
- **静态 PIE**:`-static-pie` 的 entry_va 偏移路径(B1 已覆盖)未实证,需 musl 静态 PIE smoke。
- GUI 模式下的 usability gate:目前 gate 只在 console(GUI=OFF)跑;GUI 模式的 gcc 验证靠用户手动点 Shell。
