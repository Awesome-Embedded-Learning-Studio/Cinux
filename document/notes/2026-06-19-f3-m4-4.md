# F3-M4 批4 — SIGSTOP/SIGCONT 真调度效果(TaskState::Stopped 状态机)

> 里程碑 F3-M4,2026-06-19。分支 `feat/f3-m4-scheduler`。批4 commit `e9b0dd4`。测试 836→**840/0**。

## 目标(M3 follow-up:STOP/CONT 真调度)

F3-M3 总结显式把 "SIGSTOP/CONT 真调度效果(TASK_STOPPED 状态机)留 F3-M4 调度器"。
F3-M1 的 `signal_exec_default` 对 kStop/kContinue 是 **TODO no-op**(投递了却无调度
效果)。补 `TaskState::Stopped` 状态机,让 SIGSTOP 真停、SIGCONT 真续。

## 决策

- **#1 kStop 复用 block 范式**:置 `Stopped` + `sched_class->dequeue`;若 task 即 current
  则 `Scheduler::schedule()` 切走(不返回此处直到被 SIGCONT 续)。mirrors `Scheduler::block`。
- **#2 SIGCONT/SIGKILL 在 signal_send 发送时即恢复(非投递时)**:**关键**——Stopped 任务永不被
  调度,无法自投递 SIGCONT;若只 pended,它永远醒不来。故 `signal_send` 见 Stopped 目标 +
  (SIGCONT|SIGKILL) → 立即 `Ready + enqueue`。SIGKILL 同理(须先续才能跑起来终止)。
- **#3 SIGCONT 清 pending stop**(POSIX):生成 SIGCONT 时清 pending 的 SIGSTOP/SIGTSTP/
  SIGTTIN/SIGTTOU。
- **#4 schedule() 守卫加排除 Stopped**:原「next==prev 时若 prev 非 Blocked/Dead/Zombie 则
  保留 Running」须把 Stopped 也排除——否则已停任务被重置 Running(BUG)。
- **#5 SIGKILL/SIGTERM 唤醒 Blocked 留 follow-up**:可中断睡眠(打断 futex/waitpid 阻塞)
  是更大改动,本批只处理 Stopped。signal_send 的 TODO 更新注明。

## 踩坑 / GOTCHA

- **#22 TaskBuilder().build() 消耗全局 tid 计数器,跨测试文件污染**(批4 首版教训):
  首版 stop_cont 测用 `TaskBuilder().build()` 建 victim,分到 tid 1/2/3。`run_signal_tests()`
  在 `run_scheduler_tests()` **之前**跑 → test_signal 消耗了 tid 1/2/3 → `test_scheduler.cpp:62`
  的 `test_build_basic_task` 断言首任务 `tid==1` 失败(实际 4+)。839/1。
  **解 = 用栈 `Task t{}`(同 test_sig_state 范式)**:stop/cont 只碰 state/sched_class/
  sig_actions/sig_pending,不需 TaskBuilder 的栈/ctx/slab/tid 分配。`struct Victim { Task task{}; ... }`
  栈上构造,零 tid 消耗 → 840/0。
  **通用铁律**:跨测试文件共享全局计数器(tid/pid/next_*),新测试用 TaskBuilder 建任务会
  位移他测的「首任务 tid==N」断言;纯状态机测试优先栈 Task(不分配 tid/slab/核栈)。
