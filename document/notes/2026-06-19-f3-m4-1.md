# F3-M4 批1 — SchedulingClass 策略钩子(task_tick/task_fork/task_deadline)

> 里程碑 F3-M4(调度器接口验证与增强),2026-06-19。分支 `feat/f3-m4-scheduler`(off main `5bfdbca`,M3 已合入)。
> 批1 commit `8b6c46e`。测试 827→**830/0**。

## 目标(T1:接口审查)

`SchedulingClass` 原仅 `enqueue/dequeue/pick_next/name` 四个纯虚,时间片抢占逻辑硬写在
`Scheduler::tick`(`current_slice_` >= `DEFAULT_TIME_SLICE` 即 `schedule()`)——类无发言权,
无法支持不同调度策略自拥抢占。补齐 `task_tick`/`task_fork`/`task_deadline` 三个钩子,**内聚**
时间片决策到调度类;行为向后兼容(仍每 2 滴答抢占)。

## 决策

- **#1 三个钩子全给默认实现(非纯虚)**,基类默认 `task_tick→false`/`task_fork→noop`/
  `task_deadline→0`,保证现有子类与行为零改动。RoundRobin 只 override 它需要的两个。
- **#2 时间片量子移入 RoundRobin**(`quantum_remaining_` 成员),`task_tick` 自行递减 + 耗尽
  重充 + 返回 true 请求抢占;`pick_next` 选中时重置满量子。`Scheduler::tick` 改委托给
  `current_->sched_class->task_tick()`。**删 `Scheduler::current_slice_`**(单一事实源)。
- **#3 task_tick 耗尽即重充**:若不重充,单任务系统(无别的可运行任务)`schedule()` no-op 后
  量子仍 0 → 每滴答都触发 `schedule()` 调用(浪费)。重充后下一轮又是满量子,避免空转重抢。
- **#4 task_fork 暂不接 fork/clone 调用点**:fork/clone 已 `memcpy(child,parent,sizeof(Task))`
  拷了 priority,task_fork 的"集中继承规则"今天等价 noop;真实接 fork 路径 + 优先级感知留批2
  (priority 才有意义时)。批1 只让接口完备 + 测试直接 exercise。
- **#5 task_deadline 不在 RoundRobin override**:用基类默认 0(非截止期类),给未来 RT 类预留。

## 实现

| 文件 | 改动 |
|------|------|
| [scheduler.hpp](kernel/proc/scheduler.hpp) | SchedulingClass 加 3 钩子(带默认声明);RoundRobin override task_tick/task_fork + `quantum_remaining_` 成员;Scheduler 删 `current_slice_` |

## 验证

- `timeout 40` run-kernel-test:**830/0**(+3 钩子单测)。
- 向后兼容:行为不变(2-tick 抢抢),仅时间片状态从 Scheduler 全局原子搬到 RoundRobin 类成员。

## 踩坑 / GOTCHA

- `current_slice_` 删静态定义后,4 处残留引用(init/run_first/tick/schedule)IDE 立报 undeclared
  ——逐处清,编译 830/0 证实清干净(IDE 中间快照别信,编译是真理,同 M3 批4b)。
- `task_tick` 取 `lock_.irq_guard()`:tick 在 PIT IRQ 上下文跑,enqueue/dequeue/pick_next 也都
  irq_guard(关中断)→ 不会与 tick 重入(关中断期间 PIT 不触发),锁安全。
