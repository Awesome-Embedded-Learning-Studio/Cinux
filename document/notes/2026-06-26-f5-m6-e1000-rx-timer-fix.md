# F5-M6 批b-fix — e1000 RX 时序:LAPIC timer 唤醒 sti+hlt(撤 trap 循环 hack)

> 里程碑:F5-M6(e1000 网卡驱动)。续批b(RX/TX 打通,commit 95fd10d)。
> 分支 `worktree-f5-m6-e1000`。commit `b4a846b`。
> 用户原话:「时序的问题请正确修复,千万要稳定的版本……不要拿莫名其妙的循环挡住。」

## 问题(批b 留的债)

批b 的 RX 测试(`test_arp_roundtrip` 单播 + `test_broadcast_rx` 广播)在标准
`run-kernel-test` 路径**稳定 FAIL**:`rx_dump` 显示 `GPRC=0 RDH=0`(包根本没被 MAC
投递到 ring)。批b note 却写「934/0 单播+广播收包铁证」——矛盾。

## 定位(非推测,每步有实验)

1. **复现**:target `run-kernel-test`(无 filter-dump,TCG)→ `GPRC=0` FAIL。
2. **排除 F9/SMEP/SMAP/ASLR**:注释 `enable_smep_smap()` 后照样 FAIL(rebase 到 F9
   不是元凶,尽管时间线乍看像)。
3. **加 `rx_dump` 诊断**:`GPRC=0`(MAC 没收包,不是 DMA 到 ring 失败——那种 GPRC>0)。
   → 问题在「包没到 e1000 RX 侧」,即 SLIRP 没回包 / 没投递。
4. **filter-dump pcap(决定性)**:手动 TCG + `-object filter-dump` → **全 PASS**,
   pcap 4 包双向(TX ARP request → SLIRP ARP reply → e1000 RX;DHCP 同理)。
   去掉 filter-dump → FAIL。**filter-dump 是差异**。
5. **KVM 真相**:`/dev/kvm` 是 `root:kvm`,本用户不在 kvm 组 → cmake cache 固化
   `-cpu max`(TCG,无 `-accel kvm`)。「KVM+TCG 都红」里的 KVM 从未真跑过。

## 根因

QEMU TCG 下,SLIRP 的回包是**异步投递**:要靠 QEMU **设备模型 main loop** 跑
select/poll 去 SLIRP socket 拉包、再 DMA 进 e1000 ring。

- guest 纯轮询(读 RDH 等 MMIO)**不充分驱动 main loop**;只有 vCPU 真正让出
  (`hlt`)main loop 才完整跑。
- **test kernel 设计上关中断**(确定性测试),没有周期 trap/中断把 main loop 踢起来
  → reply 永不投递 → `GPRC=0`。
- filter-dump 之所以「治好」,是它注册了 flush 定时器周期性踢 main loop——**它的副作用**;
  批b 的「读 RDH 让模型投递」结论是在**带 filter-dump 调试**时得到的,去掉就塌。
  「读 STATUS 64 次」trap 循环同理(连续 io 处理期间 main loop 顺带跑)——都是碰运气。
- production(`make run`)开中断(PIT 100Hz / LAPIC timer),main loop 周期跑,RX 正常。
  **问题仅限 test kernel。**

## 修复(硬件中断驱动,稳定)

test kernel 启用一个 **LAPIC periodic timer**(vector `0x30` = `kLapicTimerVector`,
~300Hz,复用 production divide=0x3/count=200000)+ 一个 **no-op ISR stub**(handler
只 EOI、**不调度**)。e1000 测试 poll 没包时 `sti;hlt;cli`:`hlt` 让 vCPU 让出 →
QEMU main loop 跑完投递 reply → LAPIC timer IRQ 唤醒 → 重试 poll。reply 通常 1~2 个
tick(几 ms)内就到,正常路径几乎无开销。

test kernel 全程仍 `cli`(其他 943 个测试不受影响):timer 只在 e1000 测试局部 `sti`
时才 deliver。

## GOTCHA(下一手 AI 必读,免踩坑)

### 1. 【根因】test kernel 关中断 → QEMU main loop 不跑 → SLIRP reply 不投递

模拟器上轮询驱动的 RX,**必须**让 vCPU 周期性让出(`hlt` + 周期中断)main loop 才把
入站包从后端拉进 ring。test kernel 全程 `cli` 没有这个 → `GPRC=0`。production 开中断
(PIT/LAPIC tick)天然有,所以 production 不中招。**别再写 MMIO trap 循环「泵」main
loop——那是 filter-dump 副作用级别的碰运气。**

### 2. net_timer_handler 必须 `g_lapic.eoi()`,不能 `irq_eoi(0)`

`irq_eoi()` 的 APIC/8259 dispatch 由 `switch_to_apic` 翻转;**test kernel 没调**,
所以 `irq_eoi()` 走 8259,EOI 不了 LAPIC timer(LAPIC ISR 不清 → 只唤醒一次就死)。
handler 直接 `g_lapic.eoi()`(写 EOI 寄存器 0xF0)绕过 dispatch。

### 3. handler 不调 `Scheduler::tick()`(不能干扰 test suite)

production 的 `lapic_timer_handler`(ap_main.cpp)EOI + `tick()`(抢占调度)。test 不能
复用它(tick 会扰动 943 个测试)。所以新写 `net_timer_handler`(只 EOI)。stub 用
`ISR_NOERRCODE`(EOI 由 handler 负责,和 `lapic_timer_stub` 一致)。

### 4. timer 全程 armed 但不影响其他测试

`setup_periodic_timer` 后 timer 一直跑,但 IF 关时不 deliver(IRR 累积,`sti` 时合并
deliver 一次)。只有 e1000 测试局部 `sti` 才触发。其他测试全程 `cli`,无感。

### 5. 定位法(别归因共同路径里最显眼的)

- `GPRC`(收包计数)区分「MAC 没收(filter 问题)」vs「收了没 DMA(delivery 问题)」:
  `GPRC=0` = 包没到 MAC = SLIRP 没回 / 没投递;`GPRC>0 RDH 不动` = DMA 到 ring 失败。
- `filter-dump` pcap 区分「TX 出去没」vs「SLIRP 回包没」:本例 pcap 4 包双向证明 SLIRP
  回了,问题在 e1000 RX 投递(靠 main loop)。
- 「关 X 后还 FAIL」排除法:关 `enable_smep_smap` 后仍 FAIL → F9 无关(尽管时间线像)。

### 6. KVM 权限陷阱

`/dev/kvm` 是 `root:kvm`;本用户不在 kvm 组 → `-accel kvm` permission denied。cmake
`EXISTS "/dev/kvm"` 只查文件存在不查权限,但 cache 在 sandbox 里配置时判 false → 固化
`-cpu max`(TCG)。**手动跑 QEMU 诊断时用 `-accel tcg`**,别指望 KVM。
