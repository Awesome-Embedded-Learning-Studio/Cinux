# F3-M4 批3 — 多调度类实际查询

> 里程碑 F3-M4,2026-06-19。分支 `feat/f3-m4-scheduler`。批3 commit `d57bb41`。测试 833→**836/0**。

## 目标(T3:调度类切换机制真正生效)

`register_class` 填 `classes_[class_count_]`,但 `schedule()`/`exit_current()`/`run_first()`
**直调 `default_rr_.pick_next()`,从不遍历 `classes_[]`**——第二个注册的调度类永远不会被查询,
"插拔式调度算法"名存实亡。让多类机制真正生效。

## 决策

- **#1 pick_next_from(classes, count) 公开原语**:按数组序优先级遍历,首个返回非空者胜,
  后续类仅在前面的类都空时才问。暴露为数组入参原语,**脱离全局 `default_rr_` 残留态单测**
  (测试传本地类数组,不碰 Scheduler 全局单例——避免跨测试 `default_rr_` 队列残留污染)。
- **#2 pick_next_task() 私有包装**:绑全局 `classes_/class_count_`,供 schedule/exit/run_first
  调用。三处 `default_rr_.pick_next()` 全改走它。
- **#3 注册序即优先级**(不加 class-priority 参数):`register_class(SchedulingClass*)` 签名
  不变,先注册 = 高优先级。init 先注册 default_rr_ → 普通任务最高优先级;新注册的 RT 类
  仅在 default_rr_ 空时才问。todo 草案的 `register_class(int priority,...)` 是 YAGNI(单核
  hobby os 暂只一个类 + 测试类),留后续真需要时加排序。
- **#4 向后兼容验证靠 833 回归**:生产只注册 default_rr_ → `pick_next_task()` == 旧
  `default_rr_.pick_next()`,行为不变。833 全绿佐证(schedule/exit/run_first 三个高危路径
  改写未回归)。

## 踩坑 / GOTCHA

- **全局 `default_rr_` 跨测试残留**:各测试 add_task 不成对 remove → default_rr_ 队列累积
  旧任务。若多类测试经全局 Scheduler,`default_rr_.pick_next()` 返旧任务(非空)→ 永远轮不到
  测试类,断言失败。**解 = pick_next_from 吃显式数组**,测试传本地 EmptyClass/OneShotClass
  实例数组,完全不碰全局态。这也是为何不把 pick_next_task 直接公开测(它会读全局 classes_,
  绕不开残留)。
- **本测不注册到全局 Scheduler**:EmptyClass/OneShotClass 是测函数内局部变量,pick_next_from
  收它们的数组——不进 `classes_[]`,无悬垂指针风险。
