# B2.5 — ext2 read I/O 计时观测(定位 gcc 卡顿真凶)

> 批次:perf-B2.5 · 分支 `feat/perf-b1-stats-profiling` · commit `123a6c0`(接 B2 `74ceffc`)
> 关联 memory:`gcc-compile-stutter-perf`

## 背景

B2(commit `74ceffc`)elf_load lazy 把 cc1 加载从 22s 降到 0s,gcc 编译从 31s 降到 ~8s。但 8s 还是慢。B1(84288cd+5a43320)的周期 stats 曲线显示内存子系统不是瓶颈(PMM 始终 ~2.08M 页 free、PageCache 只长到 1767 页、PF avg 1004/s)。卡在哪?B2.5 给 ext2 read 路径加 I/O 计时,直接量出占比。

## 实现

- `Ext2FileOps::read` 加 `Ext2ReadTimer` RAII —— 析构时把本次 read 的 wall time 累加到 `g_ext2_read_ns`,并 bump `g_ext2_read_count` / `g_ext2_read_bytes`。
- `dump_memory_stats` 加 `[MEM] I/O:` 行:`reads / KiB / ms`(累计)+ `(+delta since last dump)`,跟 `[MEM] #PF:`、`[MEM] PMM:` 一起按 ~1 Hz 由 stats kthread 打。

## 抓到真凶

gcc 编译 cc1 段 ext2 read I/O **5.8s / 编译 ~8s = 72%**。cc1 加载已经 0s(B2 lazy ✅),慢在 cc1 实际跑:demand PF 读 .text + header 读,ext2 **每页一次独立 AHCI 命令、无 readahead**。对照本机 Linux:顺序读 1 次 syscall 拉 32 页 + page cache 热,~1s。CinuxOS 是 per-page `read_block`(bs=1024 → 1 page = 4 block = 4 个 AHCI 命令),纯 demand。

**B3 方向定了**:readahead + 聚合 I/O,预期 5.8s → <1s。

## 附带发现

- **stats kthread 调度坑**:priority 0 + yield。不能 hlt(halt 在 band-0 拖死同级 init,init 饿死);不能太低(CPU 密集编译时被饿死,曲线断)。yield 是折中。
- **GUI ON + buildroot profile 卡**:`gui_worker` 起来后没有 EXECVE `/sbin/init`,独立 follow-up;B2.5 验证一律 GUI OFF。

## 验证

- 两 leg `run-kernel-test-all` 绿(纯观测设施,不改正确性行为)。
- 生产 boot + usability gate 跑出 I/O 曲线,72% 定位。
