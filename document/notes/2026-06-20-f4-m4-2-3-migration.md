# F4-M4 M4-2-3（修迁移 GP + AP 真跑 user task，F4-M4 收官）— 2026-06-20

> 收官 F4-M4 多核调度：根治 M4-2-2 留下的 user-task 迁移 GP（GOTCHA#25），让 AP1 真正从共享 runq 拉取并运行用户任务。分支 `feat/f4-m4-2-3-migration`（从干净 main `eb9f763` 切）。
> **根因不是 %gs 损坏**（M4-2-2 的判断被递归现场误导），而是 `GDT::load()` 用平坦数据段重载 `%gs` 选择子，在长模式下把 ap_main 刚设的 `MSR_GS_BASE` 清零。

## 背景：为什么 M4-2-2 没修成

M4-2-2 让 AP1 从 `cli;hlt` 死循环变成 `sti;hlt`，但一旦 AP1 完整 `schedule()` 迁移 user task（init/kernel_init）就**确定性 #GP**，现场 `RAX=0xF000FF53F000FF53`（非规范）。`handle_gp`→`panic`→`backtrace`+`Scheduler::current()`（读 `%gs:24`），`%gs` 一坏就递归 #GP，第一现场被污染。禁 backtrace 变干净是布局侥幸避开 race（非真修）。M4-2-2 限定 AP1 不迁移，GP 划 M4-2-3。

## 诊断：`%gs` 安全的首错捕获

`handle_gp` 递归的根因是它在碰 `%gs`（`current()`）之前没有捕获真正的 faulting RIP。本次在 [exception_handlers.cpp](../../kernel/arch/x86_64/exception_handlers.cpp) `handle_gp` **最前面**加了 `%gs` 安全的首错捕获（永久加固）：在调用 `panic()`/`current()`/backtrace 之前，用 `io_outb(0xE9,…)` 把诊断裸写到 debugcon（`build/debug.log`），全程不碰 `%gs`——frame 在栈上（经 `%rdi` 指针读）、`rdmsr` 不需要 `%gs`、`outb` 不需要 `%gs`。捕获字段：`frame->{rip,rsp,cs,err,rax,rbp}` + `cr3` + `rdmsr(GS_BASE/KERNEL_GS_BASE)`。一次性（once-flag）防递归 frame 刷屏。

**捕获到的第一次 #GP 现场**：
```
rip=<schedule() 内>  cs=0x10(内核态)  err=0
rax(prev)=0xf000ff53f000ff53   ← current() 返回的非规范垃圾指针
cr3=0x1000(内核 PML4)          ← idle/内核上下文
gs_base=0x0  kgs_base=0x0      ← 关键：GS_BASE=0(规范),不是 %gs 损坏!
```

**反汇编首错 RIP**：`movzbl 0x60(%rax),%eax` = [scheduler.cpp:306](../../kernel/proc/scheduler.cpp#L306) `if (prev->state == TaskState::Running)`。`rax=prev=current()`。GP err=0 = **非规范虚拟地址访问**（x86-64 上非规范地址产生 #GP 而非 #PF）。

**根因链**：该 CPU 在内核态（idle，cr3=内核 PML4）跑 `schedule()`，但 **GS_BASE=0**（本该是它的 percpu 块）→ `percpu()->current = %gs:24` 读地址 24（低内存 BIOS 区身份映射）→ 拿到 BIOS 远指针 `0xf000:0xff53` = `0xf000ff53f000ff53` → 非规范 → 解引用 GP。M4-2-2 看到的 `RAX=0xF000FF53...` 正是此值（之前误判为 %gs 损坏，实为下游产物）。

**为何 GS_BASE=0**：插桩在 `ap_idle_entry` 入口 dump GS_BASE = **0x0**（ap_main 设完 GS、context_switch 之后、schedule 之前）。context_switch 不碰 GS（只动 FS_BASE）。`write_msr` 正确（全 64 位）。trampoline 不碰 GS。静态全路径都说 GS 该是 percpu——矛盾点锁定在 ap_main 的 **`write_msr(GS_BASE)` 之后、`context_switch` 之前**有操作清零了 GS。

## 根因：`GDT::load()` 重载 `%gs` 清零 `MSR_GS_BASE`

[gdt.cpp:70 `GDT::load()`](../../kernel/arch/x86_64/gdt.cpp#L70)（`GDT::init()` 末尾调用）在 `lgdt` 后用平坦内核数据段选择子（0x18，基址=0）重载**所有**段寄存器，含 `%fs`（旧 L81）和 `%gs`（旧 L82）：
```asm
movw %ax, %fs
movw %ax, %gs   ← 罪魁
```

**长模式下 `mov $sel,%gs` 会用 GDT 描述符的基址覆盖 `MSR_GS_BASE`**。平坦数据段基址=0 → GS_BASE 被清零。

**ap_main 的致命顺序**（[ap_main.cpp](../../kernel/arch/x86_64/ap_main.cpp)）：L115 `write_msr(kMsrGsBase, pcpu)` 先锚定 GS=percpu → L121 `gdt_blocks[cpu_id].init()`→`load()` 把 `%gs` 重载成 0 → 之后全程 GS_BASE=0。

**BSP 为何没事**：BSP 在 boot 早期加载 GDT（`GDT::load` 也清零 %gs），**之后** `usermode_init` 才 `write_msr(GS_BASE, bsp)` 锚定，且之后不再重载 GDT。AP 顺序反了（先 GS 后 GDT）。

## 修复（Phase D）

[gdt.cpp `GDT::load()`](../../kernel/arch/x86_64/gdt.cpp#L70)：**移除 `%fs`/`%gs` 重载**（保留 `%ds/%es/%ss` + `%cs` 经 `lretq`）。

理由：长模式下 `%fs`/`%gs` 基址由 `MSR_FS_BASE`/`MSR_GS_BASE` 管理（per-thread TLS / per-CPU PerCpu），**不在 GDT 描述符里**——加载平坦选择子会强制把描述符基址(0)写进 MSR，清零锚定。null `%fs`/`%gs` 选择子 + MSR 基址是长模式 percpu 标准做法（Linux 同款）。`lgdt` 后仍需重载 `%ds/%es/%ss`（新 GDT 描述符可能不同），但 `%fs`/`%gs` 不应碰。

这是**根治**（不是 AP 顺序重排的 workaround）：消除「GDT 加载清零 GS」这个对两颗 CPU 都潜伏的脚枪。

## 启用 AP 迁移（Phase B）

[scheduler.cpp `ap_idle_entry()`](../../kernel/proc/scheduler.cpp#L106)：纯 `sti;hlt` → schedule+sti;hlt runq 拉取循环（M4-2-2 已实现、注释 gate 住的 `has_runnable_task()` 重检，去 gate）：
```c
while (true) {
    if (has_runnable_task()) { schedule(); }   // 共享 runq 有任务则切去跑
    sti; hlt;                                   // IPI 唤醒
    cli;
}
```
idle 任务（priority 255、无 sched_class）永不入 runq（schedule 的 enqueue 守卫跳过），只经 idle fallback 回到 idle；runq 只放真任务，`pick_next`-REMOVES 保证两 CPU 不双取。

- **全量 `cmake --build build`**：含 host 单测（CI 盲区），零警告通过。

## GOTCHA 更新

- **#25（M4-2-2 → M4-2-3）已闭环**：user-task 迁移 GP 根因 = `GDT::load()` 重载 `%gs` 清零 `MSR_GS_BASE`（ap_main 先设 GS 后加载 GDT），**非 %gs 损坏**（M4-2-2 现场判断被递归污染误导）。修：移除 `GDT::load()` 的 `%fs`/`%gs` 重载。教训：① 非 canonical 地址访问在 x86-64 产生 #GP(err=0) 而非 #PF；② GP handler 递归（current() 读 %gs）会污染第一现场，`%gs` 安全的首错捕获（outb/rdmsr/栈 frame，不碰 %gs）是诊断这类崩溃的正解。
- **新增 #26**：长模式下 `%fs`/`%gs` 基址由 MSR 管，**禁止用 `mov $sel,%fs/%gs` 重载**（会清零 MSR 基址）。GDT 加载后只重载 `%ds/%es/%ss`。任何未来 GDT/段初始化代码适用。
