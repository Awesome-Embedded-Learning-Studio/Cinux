## 背景:Bug ② 现象

gui_worker render 期间崩在 `fill_rect`/`glyph_blit`/`h_text`(`mov %eax,(%rdx)` 写非 canonical → #GP,或踩 .text 变 UD2 → #UD),**崩点漂移**(取决于 corrupt 指针落点)。86f2bd9 加 `InterruptGuard` 包 `desktop.render`(关 IF)作 workaround —— 全防,但副作用鼠标卡顿(IRQ 全屏蔽)。根因当时未定位,留独立弧。

## 诊断(本会话,逐步缩范围)

1. **栈高水位线探针**(host_cinux `poison_render_stack`):render 前 0x5A 灌栈底,render 后扫剩余。结果:**9916 帧全 `used=1680`(恒定),`magic_ok=1`,headroom 3800/8192**。render 调用链深度跟拖拽无关(widget 树固定),栈**根本不溢出**。「render 深栈 + IRQ frame 叠加」假设否决。
2. **撤 workaround 抓 panic**:RIP = `fill_rect` 内层 `mov %eax,(%rdx)`,**RDX = `0x00FF53F000FF5618`**(两个 staging 像素值打包成地址)。row_ptr 来自 `xrgb_row(s)` = `s.pixels + r*ppr`,row 非canonical ⇒ **s.pixels 被踩**。入口 trap 只查 nullptr/range,放过「非 nullptr 但非 canonical」的 s.pixels。
3. **h_text 入口 font canonical trap 触发**(86f2bd9 加的诊断):font 引用 = `0x458B47FF81113860`(低32位 BSS 地址 + 高32位 指令字节)。workaround 在则不触发(render 无 IRQ),撤了触发 = **render 期间的 IRQ 把 font 引用踩了**,铁证。
4. **s 移 BSS 还崩 + font corrupt** = 多个栈对象被覆盖(Surface s.pixels、Compositor font 引用、clip stack)。
5. **TCG 下动鼠标触发**(非第一帧 PIT)= 鼠标 IRQ 的 fxsave(交接 note 原说「PIT tick」是误归因,InterruptGuard 关的是全部 IF)。

## 根因(高置信度)

CinuxOS 的硬件 IRQ **没用 IST**,IRQ entry(`interrupts.S` ISR_IRQ)的 `sub $512 + fxsave (%rsp)` + ISR push frame 落在**被中断 task 的栈**(gui_worker 8KB 栈)上。PIT 100Hz / 鼠标 IRQ 在 render 中段抢进,fxsave 512B + frame 物理覆盖 render 调用链沿途的栈对象 → corrupt 指针 → fill_rect/h_text 写越界 → 踩 .text → #GP/#UD。覆盖谁取决于 IRQ 落点,所以崩点漂移。

> 之前因「栈不溢出」排除过 IST —— 错。**IST 不是防溢出,是防 IRQ frame 物理覆盖 task 栈对象**。watermark 测的是 render 自身栈(1.6KB),漏了 IRQ entry 在同栈上的 fxsave 区。font corrupt 实锤修正判断。

## 根治:IRQ 用 IST 2 栈

x86_64 IDT gate 的 `ist` 字段非 0 时,CPU 进 ISR 自动把 RSP 换成 `TSS.ist[ist-1]`,fxsave + ISR frame 落 IST 栈,**完全不碰 task 栈**。照抄 IST 1(#DF)模板:

| 文件 | 改动 |
|---|---|
| [gdt.hpp](../../kernel/arch/x86_64/gdt.hpp) | 加 `alignas(16) uint8_t irq_stack_[4KB]`(照抄 `df_stack_`) |
| [gdt.cpp](../../kernel/arch/x86_64/gdt.cpp) | `tss_.ist[1] = &irq_stack_[sizeof]`(照抄 `ist[0]`) |
| [irq_handlers.cpp](../../kernel/arch/x86_64/irq_handlers.cpp) | `irq_init` 4 处 set_handler 第5参数 `0→2`(PIC IRQ 0x20-0x2F / IPI 0xE0 / xHCI 0x40 / LAPIC timer 0x30) |
| [main.cpp](../../kernel/main.cpp) | 日志 `IST1 #DF + IST2 IRQ` |

**不改**:`interrupts.S`(CPU 自动切 IST,汇编零改)、`ap_main.cpp`(静态数组方案,AP `gdt_blocks[cpu_id].init()` 自动 init 自己的 IST 2)、`idt.hpp`(set_handler ist 参数已支持)。

per-CPU:`GDT gdt_blocks[kMaxCpus]` 每元素独立 `irq_stack_`,BSP/AP 各自 init。共 `kMaxCpus(8) × (irq 4KB + df 4KB) = 64KB` BSS。

## 对齐 Linux?

**核心思想完全对齐**:Linux x86_64 硬件中断绝不落在被中断 task 的栈上 —— 用独立硬中断栈,这正是 IST 机制。

**实现是 Linux 范式的简化版**:
- **Linux**:普通硬件 IRQ 用 **per-CPU hardirq stack**(entry 软件 切栈)+ IST 留给 NMI/#DF/#MC/#DB(这几个必须 IST,因为它们能在任意栈上下文触发,包括 CPU 正在切栈的中间态)。区分更细。
- **CinuxOS**:直接把普通 IRQ 也塞进 IST 2(省掉 entry 软件 切栈逻辑,代码更简)。效果等价(IRQ frame 不落 task 栈)。

为什么 CinuxOS 这么简化 OK:LINUX 区分 IRQ-stack vs IST 是因为 IRQ 可以嵌套在某些路径、softirq 等复杂场景;CinuxOS 调度/中断框架简单(IRQ gate IF auto-clear 不嵌套、无 softirq),一个 IST 2 给所有 IRQ 够用。**如果将来加 NMI/不可屏蔽异常,该走独立 IST 3+**(Linux 范式),不要复用 IST 2。

## ⭐ 教训 / GOTCHA

1. **「栈不溢出」≠「IRQ frame 不覆盖栈对象」**。watermark 测 render 自身栈深度(1.6KB,远不溢出),但 IRQ entry 的 fxsave 512B + frame **在同一 task 栈**(无 IST),物理覆盖 render 沿途栈对象。IST 不是防溢出,是防覆盖。
2. **静态算 fxsave 区 vs 栈对象重叠很难**(IRQ 时机 + 帧布局 + 哪层调用),耗时且易错。IST 一刀切 —— 把整个 IRQ frame 移出 task 栈,不用算。
3. **诊断 trap 的双重价值**:86f2bd9 加的 h_text font canonical trap,workaround 在则不触发(无 IRQ 不 corrupt),撤 workaround 立触发(IRQ corrupt font)—— **trap 触发/不触发本身是 IRQ-引起 的实锤**,不只是崩点定位。
4. **InterruptGuard 关全部 IF**(不只 PIT)——「PIT tick 踩栈」是误归因,真凶是任意 IRQ(鼠标实测触发)。诊断别被注释带偏。
