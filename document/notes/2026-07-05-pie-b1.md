# 2026-07-05 PIE 批1 — ET_DYN main + ASLR base 内核支持

## 背景

F9 安全弧落地了 stack/mmap/brk 三处 ASLR,但**main 可执行体的 text/data base 没随机化**——`execve` 一律 `load_elf_image(..., base=0, ...)`,假设 main 是 ET_EXEC(p_vaddr 是绝对地址)。gcc/g++ 靠 toolchain 的 `-fno-pie -no-pie`([extract.sh](../../tools/gcc-toolchain/extract.sh) + musl)强制产 ET_EXEC。

要补齐 ASLR = 让 main 支持 PIE(ET_DYN,p_vaddr base-relative),由内核选随机 base,对齐 Linux 完整 ASLR。本批 B1 做**内核侧**;B2 做 toolchain(去 `-fno-pie`,放 gcc 默认 PIE)。

handoff plan 在 `~/.claude/plans/peppy-dancing-koala.md`(已批准)。基建已就绪:`load_elf_image`([elf_load.cpp](../../kernel/proc/elf_load.cpp))早已参数化 base,`load_interpreter` 用它加载 ET_DYN ldso——**PIE main 的现成模板**。

## 实现(B1 kernel,5 文件)

### main_base 检测 + auxv/entry 偏移([execve.cpp](../../kernel/proc/execve.cpp))

```cpp
const uint64_t main_base = (ehdr->e_type == elf::ET_DYN)
                               ? (cinux::arch::USER_EXEC_BASE + cinux::lib::aslr_exec_base_offset())
                               : 0;  // ET_EXEC: p_vaddr 绝对,base 0
load_elf_image(*task->addr_space, inode.get(), ehdr, phdrs, phnum, main_base, main_img);
```

三处用 `main_base`:
- `entry_va`(static default,无 PT_INTERP 时):`main_base + ehdr->e_entry`(静态 PIE 也要偏移——**plan 漏的点**,plan 只提 AT_ENTRY)
- `aux_out->at_entry = main_base + ehdr->e_entry`(ldso 重定位 main 后跳这里)
- EXECVE 诊断日志加 `base=%p`(观测 PIE base,ET_EXEC 显示 0x0)

动态 PIE(有 PT_INTERP)`entry_va` 被 `load_interpreter` 覆盖成 ldso 入口,所以 `entry_va` 偏移只在静态 PIE 触发——但写上覆盖所有情况(ET_EXEC/ET_DYN × 静态/动态),一行不增复杂度。

### USER_EXEC_BASE([memory_layout.hpp](../../kernel/arch/x86_64/memory_layout.hpp))

```cpp
constexpr uint64_t USER_EXEC_BASE = 0x20000000ULL;  // 512 MB
```

ldso(`USER_INTERP_BASE`=256 MB)之后、mmap(`USER_MMAP_BASE`=4 GB)之前。ASLR 窗 0..16 MB → `[0x20000000, 0x21000000)`,不撞任何区。

### aslr_exec_base_offset([aslr.hpp](../../kernel/lib/aslr.hpp))

```cpp
inline uint64_t aslr_exec_base_offset() {
    return g_random.next64() & 0xFFF000ULL;  // 0 .. 16 MiB - 4K, 4 KiB-aligned
}
```

同文件 `aslr_brk_offset` 风格(4 KB 对齐、12 bit 熵)。顺手修了顶部一段过时注释(原写"non-PIE ... relocating would corrupt every absolute reference ... tracked as a separate follow-up")。

## ⭐陷阱:brk heap 与 PIE base 撞(plan 没预料)

**现象**:`run-kernel-test-all` 开 `CINUX_MUSL_DYN_SMOKE` 后,PIE `/hello-dyn` 加载即崩:
```
[EXECVE] heap VMA record failed
[PROC] execve(/hello-dyn) failed: -12
```
ET_EXEC `/hello` 仍 PASS。

**根因**:`USER_EXEC_BASE`=512 MB **大于** `USER_BRK_MAX`=240 MB。brk heap VMA 是 `insert(brk_initial, USER_BRK_MAX)`,而 `brk_initial = main_img.max_seg_end`(base 偏移后)≈ 512 MB。`[0x20005000, 0xF000000)` 是**倒置区间**(start > end)→ VMA insert 失败 → execve 返回 MapFailed(-12)。

brk heap 设计强假设 image 在低地址(`[image_end, USER_BRK_MAX)`),`USER_BRK_MAX` 是**绝对上限**(为 ldso base 256 MB 让路)。PIE main 在 512 MB,直接撞穿。

**修**:brk_max 跟随 image 位置动态化——
```cpp
const uint64_t brk_max = (main_base < cinux::arch::USER_INTERP_BASE)  // ET_EXEC 低地址
                             ? cinux::arch::USER_BRK_MAX              // 原行为(240 MB,留 ldso gap)
                             : cinux::arch::USER_MMAP_BASE;           // PIE 高地址 → 4 GB
```
- ET_EXEC 完全不变(`test_brk` 不受影响,仍用 `USER_BRK_MAX` 测 sys_brk 边界)
- PIE heap = `[brk_initial, 4 GB)`,跟随 image,不撞 ldso(256 MB)/mmap(4 GB)

最小改动,无需新常量。`USER_BRK_MAX` 保留(`test_brk` 用 + 历史文档)。

**教训**:新加载 base 要复核所有"绝对上限"假设。brk heap 的 `USER_BRK_MAX` 是为"image 永远在低地址"设计的隐含假设,PIE 打破它。类似地 mmap/stack 区也要心里有数(它们的区不依赖 image 位置,所以没事)。

## 关键决策

- **ASLR offset `0xFFF000ULL` 而非 plan 的 `0xFF0000ULL`**:plan 写 64 KB 对齐(8 bit 熵,256 位置);改成与同文件 `aslr_brk_offset` 一致的 4 KB 对齐(12 bit 熵,4096 位置)。风格统一 + 熵更大,PIE main 页对齐就够。
- **保留 `USER_BRK_MAX` 不删**:`test_brk.cpp` 用它(line 19/45/62)测 sys_brk 边界。删了破坏 host 单测,且它仍有文档价值(ET_EXEC heap ceiling)。

## 验证

- **host 产物**:`build-hello-dyn.sh` 产 `build/musl/hello-dyn`,`readelf -h` → `Type: DYN (Position-Independent Executable file)`,Entry 0x1040(base-relative),PT_INTERP + PT_PHDR 齐全。
- **`run-kernel-test-all` 两 leg 全 PASS**(单核 + -smp 2):
  - `/hello`(ET_EXEC 兼容)20/20 PASS,base=0x0
  - `/hello-dyn`(ET_DYN PIE)5/5 PASS
- **ASLR 实证**:10 次 hello-dyn execve 的 base 全不同,都落 `[0x20000000, 0x21000000)`:
  ```
  0x207C3000 0x208F3000 0x207E9000 0x20141000 0x20597000
  0x20AA5000 0x20247000 0x202AB000 0x20D10000 0x205D6000
  ```
- 无 panic/#GP/#DF/heap VMA failed。
