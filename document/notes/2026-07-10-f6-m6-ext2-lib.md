## 起因

用户要 ext2 抽象成 lib,**先于 M5 ext4 写** —— 在 host(RAM disk)跑 ext2 真逻辑 ASAN 测 + TSAN 并发测。价值:F-DYN-COV 批3/批4 在 QEMU 靠 forensics(LOCKDEP held 栈 + RaceWatchpoint + wild trace)才抓到的 `inode_cache_`/`block_alloc_` race,host TSAN 秒级抓;给 M5 ext4 写(extent 分配器)铺路(host 测快反馈,不依赖 QEMU)。

**用户决策**:路 A `libs/ext2/` 物理 lib(不走路 B 原地 host-link)。

## 调研结论(执行前)

- `inode.hpp` **完全平台无关**([inode.hpp:17-30](../../kernel/fs/inode.hpp#L17-L30)):只 include `<stdint.h>` + Cinux-Base `expected.hpp` + `fs/stat.hpp`;`Task*`/`IBlockDevice*` forward-declare(不拉 kernel 头)。lib 直接 #include 零改动。
- ext2 **不需 adapter 模式**:直接 `class Ext2 : public FileSystem`、`Ext2FileOps/Ext2DirOps : public InodeOps`;VFS 接口几乎全平台无关。
- PAL 依赖就 6 类(shim 级):kprintf/kmalloc/Spinlock/page_cache.invalidate_range/inode_ref/backtrace + **易漏的 hpet**。net_tcp host-link + host_spinlock + RAMBlockDevice 全现成。

## 关键教训
- **CinuxOS build 模型限制「物理 lib」**:build/build-host 同一份顶层 CMakeLists,无法条件 codegen → 单 STATIC target 不能同时满足 kernel(`-mcmodel=kernel`)+ host codegen。物理 lib(源码分离到 libs/)+ 双端直接编(net_tcp 范式)是 CinuxOS 的 lib 抽象方式;STATIC target 待 codegen 解。计划阶段没预见,执行撞坑后用户裁决跳过。
- **host TSAN 抓 QEMU 抓不到的 race**:lockdep 只看锁序,scratch-clobber(`block_buf_` 共享)不触发 wild block 直到时序 corrupt;host TSAN 直接看内存访问,秒级抓。M6 host-build 解锁的回归守卫价值 —— F-DYN-COV 漏的 `block_buf_` race 就是证明。
- **PAL 易漏 HPET**:ext2_common.cpp `Ext2ReadTimer` 用 `g_hpet.monotonic_ns()`(I/O 计时),host 要 `#ifndef` 守(不像 kprintf 那样显眼)。PAL 清单要 grep 全 kernel 符号(`g_hpet`/`backtrace`/`page_cache`/`inode_ref`),漏一个 → host link error。
- **`#    include` 缩进 grep 盲区**:test/unit 用 `#    include`(井号后空格缩进,条件块内),`grep '#include'`(紧连)漏命中。全面 grep 用字面路径(`'fs/ext2/'`)。
- **.gitignore `*.img` 漏测试镜像**:构建产物规则漏 commit 测试 fixture,加 `!test/data/ext2_test.img` 例外。
