# F3-M4 批2 — 优先级感知 RoundRobin

> 里程碑 F3-M4,2026-06-19。分支 `feat/f3-m4-scheduler`。批2 commit `f3b0493`。测试 830→**833/0**。

## 目标(T2:启用 priority 字段)

`Task::priority`(uint64_t,越小越优先,idle=255)此前从未被调度器使用——
`pick_next` 纯 FIFO 取队首。让它真正生效:高优先级(小值)任务优先调度,
同优先级 Round Robin。

## 决策

- **#1 选最小值,并列取最早入队者(FIFO)**:扫描就绪队列选 `priority` 最小的任务;
  并列(同优先级)取逻辑下标最小者 → 同优先级任务天然轮转。选中后 `remove_at_locked`
  紧凑环形缓冲,再尾部重入队(保持轮转语义,任务跑完不丢)。
- **#2 抽 `remove_at_locked(int i)` 私有助手**:`dequeue` 与 `pick_next` 共享环形缓冲
  紧凑逻辑,消除重复(原 dequeue 内联的 shift 循环)。
- **#3 严格优先级(可饿死)**:最高优先级任务就绪时持续被选,低优先级饿死。对齐 todo
  "简单实现"(Linux 用 nice/CFS 防饿死,留后续;hobby os 可用可调试优先)。
- **#4 不接 fork/clone 调用点**:fork/clone 已 `memcpy` 拷 priority,子进程天然继承父
  优先级,task_fork(批1)的集中规则今天等价 noop。priority 已被 pick_next 真正消费——
  这是 T2 的实质。task_fork 接 fork 路径对未来非 memcpy 的调度类才有意义,留后续。

## 踩坑 / GOTCHA

- 扫描 + 重入队的交互须手算验证:`[a5,b5,c9]` → pick a(最小下标的 pri5)→ 重入队
  `[b5,c9,a5]` → pick b → `[c9,a5,b5]` → pick a(c9 不是最小)→ ... 得 a,b,a,b,c 饿死。
  手算写入测试断言,实跑 833/0 证实。
- idle 任务 priority 255 仅信息性:idle 不在 default_rr_ 队列(schedule 的 idle fallback
  走独立 idle_task_,不经 pick_next),故 255 不参与选择。
